# 03-h2: The h^2 Heuristic

**Problem ID:** `07a-h2` -- Programming (week 3) -- h^2

## Problem Statement

Read an SAS+ file from standard input and print a single integer: the value
of the h^2 heuristic at the initial state. Print `infinity` if the goal is
unreachable.

## Background

h^m (Haslum & Geffner, 2000) generalises h^1 / h^max. Instead of tracking
a cost per single *fact* it tracks a cost per *set of up to m facts*:

```
h^m(S) = max  { cost_m(T) : T subset of S, |T| <= m }.
```

The table `cost_m[T]` is computed by fixed-point regression through every
operator.

For `m = 1` this collapses to `cost_1[f]` -- the h^1 / h^max heuristic
already implemented in `src/sasplus.rs`. For `m = 2` we additionally
track joint reachability of *pairs*:

```
cost_2[{f, g}]  =  cost to reach a state in which both f and g hold.
```

`cost_2[{f, f}]` is the single-fact cost (same as `cost_1[f]`), and in the
final reading

```
h^2(goal) = max over pairs {g1, g2} in goal of cost_2[{g1, g2}].
```

Because pairs capture interactions that single facts cannot, h^2 strictly
dominates h^1: `h^2 >= h^1` on every state, and both are admissible.

## Chain of Thought

### Step 1 -- Reuse h^1 precondition semantics

From the earlier `02-h1` and `03-hff` exercises we already learned the
correct SAS+ precondition for "effect `e` of operator `o` fires":

- every prevail condition of `o`,
- every effect's `pre_value` with `pre_value >= 0` (not just `e`'s own),
- `e`'s own conditional-effect conditions when `e` is conditional.

The same precondition is used below for regressing through each operator.

### Step 2 -- Regression over pairs of post-state facts

The central step is, for each operator `o`, enumerating every *pair of
facts the post-state of o can contain* and computing a new upper bound
on `cost_2[pair]`. There are three flavours of such pairs:

| Case | Pair source | Regression set (pre-state) |
|------|-------------|----------------------------|
| A | Both facts come from effects `e1`, `e2` of `o` (possibly the same effect, giving a singleton) | `op_pre ∪ e1.conds [∪ e2.conds if distinct]` |
| B | One fact from an effect `e`; the other persists through `o` (its variable is not touched by `o`) | `op_pre ∪ e.conds ∪ {persisting fact}` |
| C | Both facts persist through `o` | `op_pre ∪ {f1, f2}` -- useless: the regressed h^2 already contains `cost[f1][f2]` itself, so the update can never strictly lower anything. Skip. |

If the regression set is *inconsistent* (assigns two different values to
the same variable), that post-state pair cannot be achieved via this
operator; skip.

Otherwise:

```
new_cost = op.cost + h^2(regression set)
cost[p1][p2] = min(cost[p1][p2], new_cost)
```

where `h^2(S)` is the max over all ordered pairs `(p, q)` in `S x S` of
`cost[p][q]`. Singletons `(p, p)` are automatically included.

### Step 3 -- Fixed point

Initialise

```
cost[f1][f2] = 0   if both f1 and f2 hold in the initial state
             = +inf otherwise
```

Iterate the case-A/B update above until no entry changes. The process
terminates because `cost` is monotonically non-increasing and bounded
below by 0.

### Step 4 -- Goal reading

Finally,

```
h^2 = h^2(goal) = max over ordered (g1, g2) in goal x goal of cost[g1][g2].
```

Return `None` (printed as `infinity`) if the max is `+inf`.

### Step 5 -- Why this gives h^2 > h^1 in practice

On the published sample `tests/samples/01-bfs-simple/1.in` (miconic
lift-with-one-passenger), the goal is `served(p0)`. The single-fact
h^1 chain is

```
lift-at-f0 -> lift-at-f1 -> boarded(p0) -> served(p0)
```

-- cost 3 under unit costs, so `h^1 = 3`. But serving requires both
`lift-at-f0` *and* `boarded(p0)` to hold *simultaneously*, and the pair
`{lift-at-f0, boarded}` is not in the initial state. Getting it there
requires three actions under delete relaxation (`up f0 f1`,
`board f1 p0`, `down f1 f0`). h^2 captures this: `cost[lift-at-f0,
boarded] = 3`, which makes `cost[served, served] = 1 + 3 = 4`. That
matches h* = 4 -- h^2 is tight on this instance.

## Algorithm

```
offset[v] for v in 0..n  :  fact id layout, offset[v] + d indexes fact (v, d)
cost[f1][f2]             :  symmetric F x F table, initialised to
                            0 on pairs of initial facts, +inf elsewhere

repeat until no change:
  for each operator o:
    assigned[v] <- true iff o has an effect on v
    op_pre     <- prevail(o) ∪ { effect pre_values of o with pre_value >= 0 }
    if op_pre is inconsistent: skip o

    # Case A+C: both facts from effects
    for (e1, e2) with index i <= j:
      if i != j and e1.var == e2.var: skip    # post-state is inconsistent
      p1, p2 <- post-facts of e1, e2
      pre    <- op_pre + e1.conds (+ e2.conds if i != j)
      if pre inconsistent: skip
      update cost[p1][p2] with op.cost + h^2(pre)

    # Case B: one effect + one persisting fact
    for each effect e, each variable v with assigned[v] == false, each value d:
      p1 <- post-fact of e; p2 <- (v, d)
      pre <- op_pre + e.conds + {(v, d)}
      if pre inconsistent: skip
      update cost[p1][p2] with op.cost + h^2(pre)

return h^2(goal_facts)
```

Complexity: O(iters * (ops * (E^2 + E * F))), where `F` is the total
number of facts and `E` the average number of effects per operator. Each
iteration scans a full symmetric F x F table of size O(F^2). Memory is
O(F^2).

## Verification

`tests/03-h2.rs` performs two kinds of checks on every BFS sample:

1. **Dominance:** `h^2 >= h^1` (h^2 dominates h^1 by construction).
2. **Admissibility:** both `h^1 <= h*` and `h^2 <= h*`, where `h*` is
   the BFS ground-truth optimal plan length from the corresponding `.ans`
   file.

Plus one hand-traced unit test: on the problem's published sample input,
h^1 = 3 and h^2 = 4.

Measured values on the local sample set:

| Sample | h^1 | h^2 | h* |
|--------|-----|-----|----|
| `bfs-simple/1.in` | 3 | 4 | 4 |
| `bfs-simple/2.in` | 0 | 0 | 0 |
| `bfs-simple/3.in` | 2 | 2 | 2 |
| `bfs-simple/4.in` | 1 | 2 | 2 |
| `bfs-simple/5.in` | 1 | 2 | 6 |
| `BFS-medium/1.in` | 3 | 4 | 4 |
| `BFS-medium/2.in` | 1 | 1 | 6 |
| `BFS-hard/1.in`   | 3 | 4 | 4 |
| `BFS-hard/2.in`   | 3 | 5 | 7 |
| `BFS-hard/3.in`   | 5 | 9 | 15 |
| `BFS-hard/4.in`   | 7 | 8 | 127 |
| `BFS-hard/5.in`   | 1 | 2 | 6 |

h^2 is tight (equals h*) on 5 / 12 instances and strictly dominates h^1
on 10 / 12.
